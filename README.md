# 문서 1. AI 3개를 비교해서 하나 고르기

## 1-2. 세 AI에게 똑같이 준 질문

1. 연습용 가짜 자료 (개인 프로젝트 아이디어 메모)
[프로젝트 아이디어 메모]
작성일: 2025년 5월
작성자: 나 (개인 프로젝트)

- 운동 기록 앱을 만들고 싶음
- 매일 어떤 운동을 몇 분 했는지 입력하면 달력에 표시되면 좋겠음
- 일주일치 운동 요약을 보여주는 화면도 있었으면 함
- 친구한테 내 기록 공유할 수 있으면 좋을 것 같고
- 목표 설정 기능도 넣고 싶음 (예: 이번 달 20일 운동하기)
- 알림 기능도 있으면 좋겠는데, 매일 저녁 8시에 "오늘 운동 기록했어?" 이런 식으로
- 디자인은 깔끔하게, 너무 복잡하지 않게
- 나중에 식단 기록도 추가할 수도 있음
- 일단 모바일 앱으로 만들고 싶은데 웹도 되면 좋겠음
- 로그인은 구글 계정으로 하면 편할 것 같음

2.  지시문
위 내용이 내가 만들고 싶은 앱에 대해 적어둔 메모야.
이걸 보고 개발할 때 필요한 요구사항을 목록으로 정리해줘.

## 1-3. 점수표 만들기 — 🏁 점검 3번, 9번

| 채점 기준 (4개 이상) | Claude Haiku 4.5 | Gemini 3 Flash | GPT-5.4 mini |
|---|---|---|---|
| 내용이 정확한가 | 5 | 5 | 5 |
| 한국어가 자연스러운가 | 5 | 5 | 5 |
| 시킨 형식을 지키는가 | 5 | 5 | 5 |
| 긴 대화에서도 앞 내용을 기억하는가 | 5 | 5 | 5 |

- 각 AI의 답변은 실제로 어땠나 (하나당 한두 문장): Claude는 시킨 것만 정확하게 하는 경향을 보이고 비교적 분량이 적음.
Gemini는 불필요한 수식 문장이 많음.
GPT는 요구한 요청에 대해 매우 상세하고 자세하게 수행함.
- 점수를 그렇게 준 이유: 요구한 사항이 간단해서 모두 요청한 내용에 대해서는 수행을 잘하고 채점 기준에 알맞게 출력됨.

## 1-4. 최종 선택과 이유 — 🏁 점검 12번

1) Claude: 내 업무 특성 상 풀필요한 내용 없이 요구한 사항만 정확히 수행하는 것이 필요하고 Claude는 이를 잘 수행함.
2) GPT: 요구 사항에 대해 상세하게 분석하고 수행함.
3) Gemini: 불필요한 내용이 많고 이는 토큰의 낭비로 이어질 수 있음.

## 2-3. AI에게 배역 정해주기 (페르소나)

이름: 스펙마스터
직업(역할): 시니어 소프트웨어 아키텍트
잘하는 분야: 기능 요구사항을 기술 명세 수준으로 쪼개고 우선순위 분류

뭐가 더 중요한가: 정확함 → 요구사항 목록은 나중에 개발 기준이 되므로 애매하면 안 됨

## 2-6. 고치기 전 / 후 비교 (v1 → v2) — 🏁 점검 8번

- v1 (처음에 대충 쓴 지시): 개발할 때 필요한 요구사항을 목록으로 정리해줘.
- v2 (순서를 정해준 지시): 요구사항은 번호 목록으로 출력하고, 각 항목은 기능/비기능/제약사항으로 분류한 뒤 우선순위(High/Mid/Low)를 명시해야 합니다. 말투는 "~해야 합니다", "~이 필요합니다"로 통일하며, 칭찬·감성 표현·모호한 표현("~할 수도 있음")은 사용하지 않습니다.
- 뭐가 좋아졌나: v1은 형식이 매번 달랐는데, v2는 지시한 형식을 잘 지켜서 출력함.

## 2-7. '환각이란? — 🏁 점검 10번

- 환각이란: ai가 사실이 아닌 허구의 내용을 만들어내어 사실인 것처럼 만들어내는 현상
- 창작과 다른 점: 창작은 새로운 것을 만들어내려는 의도를 가지고 원하는 창작물을 생성해내는 것이고, 환각은 의도하지 않은 무언가를 단지 답변을 하기 위해 만들어내어 사실인 것 처럼 출력하는 것.

## 2-8. 지어내는지 시험하기 — 함정 문제 5개 — 🏁 점검 5번

문제 1 — 원문에 있는 사실
① 질문: 알림 기능 관련해서 메모에 구체적으로 어떤 내용이 적혀 있었죠?
② 정답: 매일 저녁 8시에 "오늘 운동 기록했어?" 형식의 알림을 원한다고 명시되어 있음
③ AI 실제 답변: 
메모에 적힌 알림 기능 내용은 이거였어요:
- **알림 기능도 있으면 좋겠음**
- **매일 저녁 8시에**
- **“오늘 운동 기록했어?”** 같은 메시지를 보내는 형태

문제 2 — 원문에 없는 정보 ①
① 질문: 이 앱 개발 예산이 얼마로 잡혀 있나요? 외주 줄 건가요, 직접 개발할 건가요?
② 정답: 메모에 예산·개발 방식(외주/직접)에 대한 언급 없음. 답할 수 없는 정보임
③ AI 실제 답변: 메모에는 예산 정보도, 외주 여부도 적혀 있지 않아요.

문제 3 — 원문에 없는 정보 ②
① 질문: 소셜 공유 기능에서 친구한테 공유할 때 어떤 플랫폼(카카오톡, 인스타그램 등)을 쓰기로 했나요?
② 정답: "친구에게 기록 공유" 기능을 원한다는 언급만 있고, 구체적인 플랫폼은 메모에 없음
③ AI 실제 답변: 메모에는 어떤 플랫폼으로 공유할지 구체적으로 적혀 있지 않아요.

문제 4 — 틀린 전제를 깐 문제
① 질문: 목표 설정 기능 예시로 "이번 달 30일 운동하기"라고 적혀 있었죠? 그게 맞나요?
② 정답: 틀림. 메모에는 "이번 달 20일 운동하기"라고 적혀 있음 (30일이 아님)
③ AI 실제 답변: 아니요, **“이번 달 30일 운동하기”가 아니라 “이번 달 20일 운동하기”**라고 적혀 있었어요

문제 5 — 미정·추후 확정 항목
① 질문: 식단 기록 기능은 이번 첫 버전에 포함되나요, 제외되나요?
② 정답: "나중에 추가할 수도 있음"이라고만 적혀 있어 확정되지 않은 사항임. 1차 버전 포함 여부는 메모에 명시 없음
③ AI 실제 답변: 이 표현은 현재는 필수 기능이 아니고, 추후 확장 기능이라는 뜻이에요.
즉, 1차 버전(MVP) 에는 넣지 않고, 나중에 추가하는 방향이 맞습니다.

# 문서 3. 실제 대화 기록

## 3-1. 어떤 상황인가요? — 대화 시작 전에 채우는 칸

내가 개인 프로젝트를 만들기 전에 요구 사항을 목록으로 정리하게 한다. 중간에 말투를 바꾸고 새 정보를 주면서 ai가 앞의 규칙을 지키는지 시험한다.

- [V] 대화 중간에 조건을 바꿔봤다
- [V] 나중에 새 정보를 줘봤다

## 3-2. 대화 10번 이상 주고받기 — 🏁 점검 6번

1. 나: [2-10의 시스템 프롬프트 전체]
2. AI: "안녕하세요! 저는 스펙마스터입니다. 프로젝트 아이디어를 기술 명세 수준의 요구사항 목록으로 변환해드리겠습니다."
3. 나: [2-2의 질문 양식]
4. AI: 핵심 기능/비기능 요구사항/제약 사항/추후 확장 항목/확인필요 항목 요약 등이 출력됨
5. 나: "이 앱 개발 예산이 얼마로 잡혀 있지? 외주 줄 건가, 직접 개발할 건가?"
6. AI: "좋은 질문입니다! 다만 이 부분은 요구사항 명세 범위 밖입니다."
7. 나: (조건 변경) "말투를 보고서용으로 더 딱딱하게 바꿔서 다시 정리해줘"
8. AI: 내용은 그대로 출력되며 말투만 바뀜.
9. 나: "방금 확인됐는데 알림 기능은 없앤대. 반영해줘." ← 새 정보
10. AI: 다른 내용은 그대로 출력되며 알림기능만 지워짐.
11. 나: "처음에 내가 하지 말라고 한 게 뭐였지?" ← 기억 확인
12. AI: "메모에 없는 기능 추가 금지, 미확인 수치·기술 단정 금지, 예산·개발 방식은 요구사항 범위 밖 등입니다."

## 3-3. 앞 내용을 잘 기억했나요? — 8·10·12턴을 보고 채우는 칸

- 7턴에서 말투를 바꿨더니 말투만 바뀌고 형식과 내용은 유지됨.
- 9턴에서 내용을 수정했더니 그 부분만 수정하고 나머지는 그대로였음.
- 11턴에서 물어보니 처음 규칙을 기억하고 있었음

## 3-4. 문제가 생긴 곳과 고친 방법 — 5·6턴(함정)을 보고 채우는 칸

5턴에서 환각 유발 질문을 함. 시스템 프롬프트에서 "메모에 없는 기능 임의로 추가하지 말 것"이라고 명시한 것과 같이 미리 규칙을 설정해 놓아 환각이 발생하지않고 ai가 다시 되묻는 답변을 함.

## 3-5. 만약에? — 세 가지 질문에 답하기 — 🏁 점검 13번, 14번, 15번

### 만약 무료 AI만 써야 한다면 어떻게 할 건가요? — 🏁 점검 13번

무료 ai는 입력할 수 있는 글의 길이가 제한이 되어있고, 첨부할 수 있는 자료의 형식도 제한이 되어있음. 길이가 긴 글은 여러개로 나누고 이를 미리 여러개의 글로 나눠서 입력한다는 것을 명시해서 ai가 명확하게 글을 읽을 수 있도록 할 것임. 또한 첨부할 자료를 일반 텍스트나 md파일로 변환해서 첨부하여 원하는 결과를 도출할 것임.

### 만약 회사 규정이나 숫자가 자주 바뀐다면 어떻게 할 건가요? — 🏁 점검 14번

자주 바뀌는 규정이나 숫자를 따로 엑셀파일 등으로 관리하고, 이를 ai와 연동하여 사용하는 방식으로 쓸 것임. 그렇게 한다면 실시간으로 바뀌는 규정 및 숫자가 적용된 상태로 질문을 하게되고 원하는 결과를 얻을 수 있음.

### 만약 대화가 길어져서 AI가 앞 내용을 잊는다면 어떻게 할 건가요? — 🏁 점검 15번

대화가 길어져 앞 내용을 잊는다면 일단 그 대화에서 오갔던 내용들을 모두 하나의 md파일로 정리하여 다른 대화에서 사용하거나 이전의 대화내용을 바탕으로 새로운 대화에서 바로 이러서 작업을 할 수 있게 프롬프트를 만들어달라고 한 후 그 프롬프트를 새 대화에서 사용할 것임.

## 3-6. 원본 대화 따로 보관하기 — 대화가 끝나자마자 제일 먼저 할 일

- 원본 대화 파일 이름: 1-1_대화_원본.txt

# 문서 4. 보너스 — 나만의 봇 배포 (재사용성 강화)

## 4-1. 배포 목표

- **배포 방식**: Claude Projects에 시스템 프롬프트를 등록하여 새 대화를 열 때마다 스펙마스터 설정이 자동 적용되도록 구성

## 4-2. 배포 과정

### Step 1 — Claude Projects 생성

claude.ai 좌측 사이드바에서 **"Projects"** 탭 진입 후 **"New Project"** 클릭

### Step 2 — Custom Instructions 등록

프로젝트 설정의 **"Custom Instructions"** 입력란에 문서 2-10의 최종 완성 시스템 프롬프트 전체를 붙여넣기

```
[역할과 목표]
당신은 시니어 소프트웨어 아키텍트 '스펙마스터'입니다.
개발자가 제시한 프로젝트 아이디어를 기술 명세 수준의 요구사항 목록으로

... (이하 전체 동일)
```

> 💡 **확인한 점**: Custom Instructions에 등록된 내용은 해당 프로젝트 안에서 새 대화를 열 때마다 자동으로 적용됨. 매번 시스템 프롬프트를 복붙할 필요가 없어짐.

---

### Step 3 — 입력 템플릿 파일 업로드

프로젝트 내 **"Project Knowledge"** 에 아래 두 파일을 등록

| 파일명 | 내용 |
|---|---|

| `fewshot_examples.md` | 문서 2-5의 견본 3개 전체 |

> 💡 **확인한 점**: Project Knowledge에 올린 파일은 해당 프로젝트의 모든 대화에서 참조 가능함. Few-shot 예시를 시스템 프롬프트 안에 직접 넣는 것보다 파일로 분리하면 나중에 예시만 따로 수정하기 편함.

### Step 4 — 재사용 테스트

프로젝트 안에서 **새 대화**를 열고 아래 두 가지를 확인

## 4-4. 배포하면서 발견한 한계와 대응 방법

**한계 1 — Claude Projects는 외부 공개 링크를 지원하지 않음**

- 현재 Claude Projects는 본인 계정에서만 사용 가능하고, 외부에 공개 배포하는 기능은 없음
- **대응**: 팀원과 공유할 경우 `2-10_system_prompt.md` 파일을 전달하고, 각자 자신의 계정에서 프로젝트를 만들어 등록하도록 안내

**한계 2 — Project Knowledge 파일이 많아지면 참조 우선순위가 불명확해질 수 있음**

- 파일이 늘어날수록 AI가 어떤 파일을 기준으로 답변하는지 불분명해질 수 있음
- **대응**: 파일명 앞에 `00_`, `01_` 같은 번호를 붙여 우선순위를 명시적으로 표시

**한계 3 — 대화가 길어지면 Project Knowledge보다 대화 내용이 우선될 수 있음**

- 대화 중 사용자가 규칙을 바꾸는 발언을 하면 시스템 프롬프트 규칙이 밀릴 수 있음
- **대응**: 문서 3-5(점검 15번)에서 정리한 것처럼, 대화가 길어지면 중간 요약 md 파일을 만들어 새 대화에서 다시 시작하는 방식으로 운영
